## TL;DR

`using Eigen::Jacobian;` fails because **the header that defines `Eigen::Jacobian` is not (or cannot be) included** in your translation
unit, or you are using a **very old Eigen version** that does not contain that symbol.

The fix is usually one (or more) of the following:

| # | What to do | Why it works |
|---|------------|--------------|
| 1️ ⃣ | **Make sure you include the right Eigen header *before* the `using` line**<br>```cpp\n#include <Eigen/Core>   // or #include
<Eigen/Dense>\n``` | `Eigen::Jacobian` (the *class* in `Eigen/Core`/`Eigen/Dense`) is declared in `Eigen/Core` (or any higher‑level Eigen
header that pulls `Core` in). If you only include `<Eigen/Geometry>` or a custom header that forwards‑declares a `Jacobian` function, the
name will be missing. |
| 2️ ⃣ | **Upgrade Eigen** if you are on a version older than ~3.3.0. | The `Jacobian` *class* was introduced in `Eigen/Core` around 3.3. In
older releases you will only find a *function* `Eigen::Jacobian(const Expr&)` that returns a matrix, not a `using Eigen::Jacobian;` alias.
Upgrading or adding `#include <Eigen/Unsupported/Utils/Eigen-Jacobian.h>` can help. |
| 3️ ⃣ | **If you are trying to use the AD‑friendly `Jacobian` *function* (not the class)**<br>Replace the line with `#include
<Eigen/Autodiff>"` or `using Eigen::Jacobian; // pulls in the autodiff namespace` *or* write `Eigen::Jacobian<YourExpr>` directly. | The AD
module lives in `unsupported/Eigen/AD`. The plain `using Eigen::Jacobian;` works only after you have included
`Eigen/unsupported/Utils/Eigen-Jacobian.h`. |
| 4️ ⃣ | **Check your include order** – any `#include` that pulls in a forward‑declared `Jacobian` *must* be *after* the `using
Eigen::Jacobian;` line. | Forward declarations are fragile: if you wrote `using Eigen::Jacobian;` first, the compiler will look for the
symbol before it has seen its definition. Put the `#include` **above** the `using` line. |
| 5️ ⃣ | **Make sure the compiler can actually find Eigen**. Add to your compiler command line: `-I/path/to/eigen` (or set the include path in

## 1️ ⃣ Deep dive – Where does `Jacobian` actually live?

Eigen contains *three* distinct ways to talk about a Jacobian:

| Symbol | Where it lives | What it *is* | Typical use |
|--------|----------------|--------------|--------------|
| `Eigen::Jacobian<Expr>` | **`<Eigen/Core>`** (or any higher header) | A **class template** that stores a lazily‑evaluated Jacobian of an
expression. | Used when you want to *build* a Jacobian matrix once and reuse it (e.g. in a `ceres::Manifold` or a custom optimizer). |
| `Eigen::Jacobian(const Expr&)` | **`<Eigen/Core>`** (a *function* overload) | Returns a **temporary matrix** containing the numerical
Jacobian of a function call. | Handy for one‑shot Jacobian evaluation in tests or scripts. |
| `Eigen::Jacobian` (the *namespace alias* you imported) | **`<Eigen/unsupported/Utils/Eigen-Jacobian.h>`** (AD module) | A **type‑alias**
that forwards to `Eigen::Jacobian<...>` *in the AD namespace* (`Eigen::auto_diff`). | Needed when you use Eigen’s **automatic
differentiation** (`.derivative()`) utilities. |

`curve_fit.hpp` in the **Ceres Solver** source (the file that gave you the error) expects the **first** variant (the class). The original
author wrote:

```cpp
using Eigen::Jacobian;      // pulls in Eigen::Jacobian<Expr>
```

That line **only works** when `<Eigen/Core>` (or a higher header) has been included **earlier**, and **only if your Eigen version ≥ 3.3**.
The error you saw means the compiler never saw the declaration of that class.

## 2️ ⃣ How to verify your Eigen version

Run a quick compile‑time check (or just open a header). The version string is in `Eigen/src/Core/Common.h`:

```cpp
#if EIGEN_WORLD_VERSION != 3 || EIGEN_MAJOR_VERSION < 3
#error "Eigen 3.x is required"
```

You can also print it from the command line:

```bash

```

If you get something older than `3.3.0` you should **download the latest** (3.4.x is the current stable at the time of writing) from
https://gitlab.com/libeigen/eigen. The installation is header‑only: just unpack somewhere and add that directory to your compiler’s include
path.

## 3️ ⃣ Common pitfalls that lead to the error

| Symptom | Likely cause | Quick fix |
|---------|--------------|-----------|
| `Jacobian has not been declared` after `#include <Eigen/Geometry>` only. | `Eigen/Geometry` *does* include `Eigen/Core`, but some old
Eigen releases used a *forward* declaration that was later moved to `Core`. | Replace `#include <Eigen/Geometry>` with `#include
<Eigen/Dense>` (or explicitly `#include <Eigen/Core>`). |
| The line `#include <curve_fit.hpp>` appears *before* any Eigen header in a source file. | The `using` inside `curve_fit.hpp` is processed
before your source file pulls in Eigen. | Move *all* your Eigen includes to the top of the source file (or at least before any `#include
"curve_fit.hpp"`). |
| You are compiling with **`-DEIGEN_NO_AUTODIFF`** (or similar) in the project's config. | The AD module is stripped, and the `Jacobian`
class is disabled. | Remove the macro or define `EIGEN_AUTODIFF_NO_EXPORT` after the Eigen includes (but this is rarely needed). |
| You have **two copies of Eigen** on your system (one in `/usr/include/eigen3` and another in your project). | The file that gets parsed
first may be a stale copy that does not have the Jacobian class. | Ensure there is **only one** copy on the include path, or use `-I` with
explicit precedence. |
| You are using **`-fno-exceptions`** and the Jacobian class has a constructor that throws. | Not a *declaration* error, but a *link* or
*compile* error you may see as “has not been declared”. | Build Eigen without exception‑heavy features (`-DEIGEN_NO_EXCEPTIONS`) if you
cannot enable exceptions. |

3. **Check your compiler command line** – make sure the `-I` points to the *directory* that contains the `Eigen` folder (e.g.

   ```bash
   g++ -v -E -x c++ - <<'EOF'

   int main(){}
   EOF
   ```

   will show you which include directories are searched. Verify that the *first* one points to the correct Eigen tree.

4. **Re‑compile**. If the error persists, clean all object files (some IDEs keep stale caches) and rebuild.

5. **If you still get the same error** – run the preprocessor to see what the compiler actually sees:

   ```bash
   g++ -E -I/path/to/eigen -dM - < /dev/null | grep Jacobian
   ```

   You should see something like:

   ```

   ```

   If the line is missing, the header containing `Jacobian` was *not* seen.

## 5️ ⃣ When you really *need* the AD‑Jacobian

If your code uses `ceres::LocalParameterization::Jacobian` that works with **automatic differentiation** (e.g., `ceres::Jet`), you will need
the **AD version** of `Jacobian` from `unsupported/Eigen/AD`. In that case:

```cpp
#include <Eigen/Dense>
#include <Eigen/unsupported/Utils/Eigen-Jacobian.h>   // <-- pulls in AD Jacobian class
using Eigen::Jacobian;   // this now resolves to Eigen::Jacobian<YourExpr, AD_SPECIFIC>
```

You may also need to define a macro **before** including the AD header:

```cpp
#define EIGEN_NO_AUTODIFF  // if you *don't* want AD
// or
#define EIGEN_AUTODIFF
```

But for most Levenberg‑Marquardt applications that do **not** use AD, the plain `Eigen::Jacobian<Expr>` class is sufficient, and the simple
`#include <Eigen/Dense>` is all you need.
